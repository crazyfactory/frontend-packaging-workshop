Benefits of package

  • In case there's a bug, it's easier to revert the version number
  • Changelogs for management helps them see clearly and in a wide scope
  • Seperation of concern
  • Re-Usability

Benefits for developer

  • Separate/Isolated development
  • Updating your dependencies and not having to worry if your project breaks someone else's
  • Control over which tool/technology you want to use
  • Automation and separate repository access, more controlled styleguide.
  • Separate linting, building and not having to deal with legacy code, ever!
  • Use latest features from frontend which might include scss and es2017 (new javascript standard)
  • Use any frontend technology, react, angular, vue, you name it.
  • Checking code quality and testing is faster if there are multiple smaller packages

Disadvantages of packages

  • Longer feedback cycle
  • Requires maintenance across multiple repositories
  • For private repositories it might mean a tiny bit of added cost

Quality of good package

  • Easy to setup
  • Easy to get started for any changes
  • CI integration
  • Less things to setup
  • Less dependencies, and every of them should just come with a npm install
  • Easy to develop

Publishing a package

  • if you're using any new feature, you should transpile your package
  • create a .npmignore file (npm ignores from .gitignore by default)
  • check main entry point in package.json
  • Check homepage entry in package.json
  • Once everything is done, simply run 
    npm run publish
  • I have forgotten to update version number in package.json
  • I also did forget tag commit with version number
  • I also have forgotten to write a change log
  • I'm also lazy to do this once in a while

Semantic Release

  • Runs every time when something is merged in default branch (master in most of the cases)
  • Since this is automated, there's no way it'd forget
  • It's stable
  • Once something is merged, it generates change log for that release
  • also comments on PR and issues and tells which version this issue is resolved in
  • Creates actual releases on github instead of normal tagged commits
  • Saves a lot of developer's time

Helping semantic release

  • always use angular style commit message
  • setup hooks to prevent people from writing with invalid commit message
  • Reject PRs which have completely random commit messages

Husky

git hooks made easy 


"husky": {
    "hooks": {
      "commit-msg": "validate-commit-msg",
      "pre-commit": "npm run lint",
      "pre-push": "npm run build && npm run build:prod && npm test"
    }
  }

Greenkeeper

Creates PR for any packages which have a new update

Translation

Manage yourself, leverage something else to do actual translation. Have single source of truth on your package

Tips

  • Don't be afraid to try things out
  • Enforce as strict linting as possible
  • If there's a style contributor uses but you don't like, feel free to write a lint rule
  • Automate, anything you did for second time, see if automation is possible
  • Trust your builds, but not blindly
  • See if you can opensource parts of your project, this cuts down costs
